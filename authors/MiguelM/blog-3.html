Sequencing Activities

19 February, 2022

Within Project Schedule Management, there is a step known as ‘Sequencing Activities’. This step is very important as it sets the pace for a project and the activities it involves. Setting an order to these activities also involves figuring out what their dependencies are. An author defines dependencies as “a relationship between two modeling elements, in which a change to one modeling element (the independent element) will affect the other modeling element (the dependent element)” (Wideman, 2017). The importance of understanding dependencies when choosing the order in which to perform activities within a project cannot be understated. Since modifying an independent element directly affects the dependent element in the same scope, a failure to understand how the two are connected will invariably have a negative effect on the outcome. Furthermore, another important reason for understanding dependencies between different activities is the fact that some activities need to be completed before others can be started. In some cases, some activities can be done in parallel, while some others may show a degree of overlap depending on the nature of the activity. Among the different types of existing dependencies, there are four that stand out the most as it relates to their respective reliance upon other activities within a project. The four dependencies are: Finish-to-start, Start-to-start, Finish-to-finish and Start-to-finish (airfocus). The Finish-to-Start dependency is usually the most recurring relationship between two different tasks. In this type of dependency, the previous task must be completed before the next task can begin. Another type of dependency is Start-to-start. This type of dependency requires the previous task to have started before the next dependency can start, and the finish type for either activity is not a factor. The next dependency is the Finish-to-finish type. In this dependency, the following task can only finish after the previous task has been completed. The start time of either activity is not a factor in this dependency. The last type of dependency is the Start-to-finish dependency. This dependency is the least recurring one between different tasks. This dependency requires the following activity to have already started and cannot be completed before the previous activity has been started also. Project dependencies are necessary for a variety of reasons, and it order to understand the reasoning behind the sequencing of the activities within a project, the need for these dependencies must also be understood. First off, Agile Scrum Masters must decide the order of these sequences of activities in the context of the project plan. Secondly, project managers must devise a tasks’ critical path in order to understand the length of time a given activity will take to complete. Another reason for dependencies is that managers also have the responsibility of identifying required resources to finish certain tasks, at the same time that they have to deal with possible scheduling conflicts. The next reason for dependencies is the fact that managers need to monitor and assign tasks that form the overarching project plan. One last reason project dependencies are necessary is because Scrum Masters must also identify and act on any chance that comes by to decrease the time it takes to complete each task in the project schedule. One of the best tools to represent activity sequencing is network diagrams. A network diagram is “a graphical portrayal of the activities and events of a project” (AcqNotes,2021). Its role is to display the way in which each activity within a project relates to others, and the order of precedence between these. Another advantage of using these diagrams is that they make determining a project’s late starts and finishes easier, while also providing insight into how resources are being utilized and further allowing project managers the opportunity to create hypothetical scenarios in order to be prepared for the future. This wealth of information enables managers to oversee the status of given plan or process and evaluate alternate routes of progress. There are several benefits to be reaped from implementing a network diagram. One of these is that these diagrams provide a manager with the ability to track every element of the project and enables them to better understand its current status. Another benefit is the ability to communicate project progress to the pertinent stakeholders. Also, these diagrams make it easy to design project workflows and use them to track different types of dependencies within a project’s tasks. Lastly, these diagrams also make it easier for managers to accurately represent project bottlenecks which could potentially become a problem to the project’s overall progress.
